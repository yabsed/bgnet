5 시스템 호출로 승부하기

이 절에서는 유닉스 머신, 또는 소켓 API를 지원하는 어떤 시스템이든(BSD, Windows, Linux, Mac 등등) 그 네트워크 기능에 접근하게 해 주는 시스템 호출과 기타 라이브러리 호출을 살펴봅니다. 이런 함수들 중 하나를 호출하면 커널이 넘겨받아 마법처럼 일을 처리해 줍니다.

대부분의 사람들이 여기서 막히는 지점은 이것들을 어떤 순서로 호출해야 하는가입니다. 아마 이미 느끼셨겠지만, 이런 점에서는 man 페이지가 별 도움이 안 됩니다. 그래서 그 끔찍한 상황을 조금이라도 덜어 보기 위해, 아래 절들에서 시스템 호출을 여러분 프로그램에서 필요한 순서와 정확히(대략) 같은 순서로 배치해 보았습니다.

여기에 군데군데 들어 있는 샘플 코드 몇 조각, 우유와 쿠키(이건 아마 직접 준비하셔야겠지만), 그리고 약간의 배짱과 용기만 더해지면 인터넷 위로 데이터를 존 포스텔의 아들처럼 쏘아 보낼 수 있을 겁니다!

(간단히 보여 주기 위해 아래 코드 조각 중 많은 부분은 필요한 에러 검사를 포함하지 않습니다. 또 getaddrinfo() 호출이 성공해서 연결 리스트 안에 유효한 항목을 돌려준다고 가정하는 경우가 아주 흔합니다. 하지만 독립 실행 예제 프로그램에서는 이 두 상황을 모두 제대로 다루고 있으니, 그쪽을 모델로 삼으세요.)

5.1 getaddrinfo()—출발 준비!

옵션이 많아 진짜 일꾼 같은 함수지만, 실제 사용법은 꽤 단순합니다. 나중에 필요한 struct를 준비하는 데 도움을 줍니다.

짧게 역사를 보자면, 예전에는 DNS 조회를 하려면 gethostbyname()이라는 함수를 사용했습니다. 그런 다음 그 정보를 직접 struct sockaddr_in에 손으로 채워 넣고 호출에 사용했죠.

다행히 이제는 그럴 필요가 없습니다. (IPv4와 IPv6 모두에서 동작하는 코드를 쓰고 싶다면, 사실 그렇게 하는 게 바람직하지도 않죠!) 현대에는 getaddrinfo()가 있어서 DNS 조회, 서비스 이름 조회, 필요한 struct 채우기까지 온갖 좋은 일을 다 해 줍니다.

한번 봅시다!

#include <sys/types.h>
#include <sys/socket.h>
#include <netdb.h>

int getaddrinfo(const char *node,   // e.g. "www.example.com" or IP
                const char *service,  // e.g. "http" or port number
                const struct addrinfo *hints,
                struct addrinfo **res);

이 함수에 입력 파라미터 세 개를 넘기면, 결과가 담긴 연결 리스트 res의 포인터를 돌려줍니다.

node 파라미터는 연결하려는 호스트 이름 또는 IP 주소입니다.

다음은 service 파라미터인데, "80" 같은 포트 번호일 수도 있고, "http", "ftp", "telnet", "smtp" 같은 특정 서비스 이름일 수도 있습니다. 이런 이름은 IANA 포트 목록21이나 유닉스 시스템의 /etc/services 파일에서 찾을 수 있습니다.

마지막으로 hints 파라미터는 관련 정보를 미리 채워 넣어 둔 struct addrinfo를 가리킵니다.

아래는 서버가 자신의 호스트 IP 주소의 3490번 포트에서 대기하려고 할 때의 예제 호출입니다. 실제로 listen()을 하거나 네트워크 설정을 하지는 않고, 나중에 쓸 구조체만 준비합니다:

int status;
struct addrinfo hints;
struct addrinfo *servinfo;  // will point to the results

memset(&hints, 0, sizeof hints); // make sure the struct is empty
hints.ai_family = AF_UNSPEC;     // don't care IPv4 or IPv6
hints.ai_socktype = SOCK_STREAM; // TCP stream sockets
hints.ai_flags = AI_PASSIVE;     // fill in my IP for me

if ((status = getaddrinfo(NULL, "3490", &hints, &servinfo)) != 0) {
    fprintf(stderr, "gai error: %s\n", gai_strerror(status));
    exit(1);
}

// servinfo now points to a linked list of 1 or more
// struct addrinfos

// ... do everything until you don't need servinfo anymore ....

freeaddrinfo(servinfo); // free the linked-list

ai_family를 AF_UNSPEC으로 둔 것에 주목하세요. 즉 IPv4를 쓰든 IPv6를 쓰든 상관없다는 뜻입니다. 하나를 특정하고 싶다면 AF_INET이나 AF_INET6로 설정하면 됩니다.

또 AI_PASSIVE 플래그가 보일 텐데, 이건 getaddrinfo()에게 소켓 구조체에 로컬 호스트 주소를 채워 넣으라고 알려 줍니다. 그래서 주소를 하드코딩하지 않아도 되니 편리하죠. (또는 위 코드에서 지금 NULL이 들어간 첫 번째 인자 자리에 특정 주소를 넣어도 됩니다.)

이제 호출을 합니다. 오류가 발생하면(getaddrinfo()가 0이 아닌 값을 반환하면) 보이는 것처럼 gai_strerror() 함수로 오류를 출력할 수 있습니다. 반대로 모든 것이 잘 되면 servinfo는 struct addrinfo들의 연결 리스트를 가리키게 되고, 각 원소에는 나중에 사용할 수 있는 어떤 종류의 struct sockaddr가 들어 있습니다. 멋지죠!

마지막으로, getaddrinfo()가 친절하게 할당해 준 연결 리스트를 더 이상 쓰지 않게 되면, freeaddrinfo()를 호출해 전부 해제할 수 있고 또 그래야 합니다.

이번에는 클라이언트가 특정 서버, 예를 들어 "www.example.net"의 3490번 포트에 연결하고 싶을 때의 예제입니다. 역시 실제로 연결을 하지는 않고, 나중에 쓸 구조체만 준비합니다:

int status;
struct addrinfo hints;
struct addrinfo *servinfo;  // will point to the results

memset(&hints, 0, sizeof hints); // make sure the struct is empty
hints.ai_family = AF_UNSPEC;     // don't care IPv4 or IPv6
hints.ai_socktype = SOCK_STREAM; // TCP stream sockets

// get ready to connect
status = getaddrinfo("www.example.net", "3490", &hints, &servinfo);

// servinfo now points to a linked list of 1 or more
// struct addrinfos

// etc.

계속 servinfo가 온갖 주소 정보를 담은 연결 리스트라고 말하고 있죠. 이 정보를 보여 주는 짧은 데모 프로그램을 하나 써 봅시다. 이 짧은 프로그램22은 명령줄에서 지정한 호스트의 IP 주소를 출력합니다:

/*
** showip.c
**
** show IP addresses for a host given on the command line
*/

#include <stdio.h>
#include <string.h>
#include <sys/types.h>
#include <sys/socket.h>
#include <netdb.h>
#include <arpa/inet.h>
#include <netinet/in.h>

int main(int argc, char *argv[])
{
    struct addrinfo hints, *res, *p;
    int status;
    char ipstr[INET6_ADDRSTRLEN];

    if (argc != 2) {
        fprintf(stderr,"usage: showip hostname\n");
        return 1;
    }

    memset(&hints, 0, sizeof hints);
    hints.ai_family = AF_UNSPEC;  // Either IPv4 or IPv6
    hints.ai_socktype = SOCK_STREAM;

    if ((status = getaddrinfo(argv[1], NULL, &hints, &res)) != 0) {
        fprintf(stderr, "getaddrinfo: %s\n", gai_strerror(status));
        return 2;
    }

    printf("IP addresses for %s:\n\n", argv[1]);

    for(p = res;p != NULL; p = p->ai_next) {
        void *addr;
        char *ipver;
        struct sockaddr_in *ipv4;
        struct sockaddr_in6 *ipv6;

        // get the pointer to the address itself,
        // different fields in IPv4 and IPv6:
        if (p->ai_family == AF_INET) { // IPv4
            ipv4 = (struct sockaddr_in *)p->ai_addr;
            addr = &(ipv4->sin_addr);
            ipver = "IPv4";
        } else { // IPv6
            ipv6 = (struct sockaddr_in6 *)p->ai_addr;
            addr = &(ipv6->sin6_addr);
            ipver = "IPv6";
        }

        // convert the IP to a string and print it:
        inet_ntop(p->ai_family, addr, ipstr, sizeof ipstr);
        printf("  %s: %s\n", ipver, ipstr);
    }

    freeaddrinfo(res); // free the linked list

    return 0;
}

보시다시피 이 코드는 명령줄에서 받은 값을 가지고 getaddrinfo()를 호출하고, 그 결과로 res가 가리키는 연결 리스트가 채워집니다. 그런 다음 리스트를 순회하면서 출력하거나 원하는 일을 할 수 있습니다.

(IP 버전에 따라 서로 다른 struct sockaddr 타입 안으로 파고들어야 하는 약간 못생긴 부분이 있습니다. 미안해요! 저도 이걸 더 우아하게 피하는 방법은 잘 모르겠습니다.)

샘플 실행입니다! 다들 스크린샷을 좋아하잖아요:

$ showip www.example.net
IP addresses for www.example.net:

  IPv4: 192.0.2.88

$ showip ipv6.example.com
IP addresses for ipv6.example.com:

  IPv4: 192.0.2.101
  IPv6: 2001:db8:8c00:22::171

이제 이 부분은 정리됐으니, getaddrinfo()에서 얻은 결과를 다른 소켓 함수들에 넘겨 마침내 네트워크 연결을 맺어 보겠습니다! 계속 읽어 주세요!

5.2 socket()—파일 디스크립터를 얻자!

이제 더는 미룰 수 없겠네요. socket() 시스템 호출을 이야기해야 합니다. 형태는 이렇습니다:

#include <sys/types.h>
#include <sys/socket.h>

int socket(int domain, int type, int protocol);

그런데 이 인자들은 뭘 의미할까요? 어떤 종류의 소켓을 원하는지(IPv4인지 IPv6인지, 스트림인지 데이터그램인지, TCP인지 UDP인지) 말하게 해 줍니다.

예전에는 이런 값을 하드코딩하는 일이 흔했고, 지금도 물론 그렇게 할 수 있습니다. (domain은 PF_INET 또는 PF_INET6, type은 SOCK_STREAM 또는 SOCK_DGRAM, protocol은 주어진 type에 맞는 프로토콜을 고르도록 0으로 둘 수 있습니다. 또는 getprotobyname()을 호출해 "tcp"나 "udp" 같은 프로토콜을 조회할 수도 있습니다.)

(이 PF_INET은 struct sockaddr_in의 sin_family 필드를 초기화할 때 쓰는 AF_INET과 아주 가까운 친척입니다. 실제로 둘은 너무 가까워서 값도 같습니다. 그래서 많은 프로그래머가 socket() 첫 번째 인자로 PF_INET 대신 AF_INET을 넘기곤 하죠. 자, 우유와 쿠키를 준비하세요. 이야기를 할 시간입니다. 아주 오래전 옛날 옛적에는 주소 패밀리(즉 "AF_INET"의 "AF") 하나가, 프로토콜 패밀리(즉 "PF_INET"의 "PF")라고 불리는 여러 프로토콜을 지원할지도 모른다고 생각했습니다. 하지만 그런 일은 일어나지 않았습니다. 그리고 모두 행복하게 살았습니다. 끝. 그러니 가장 정확한 방법은 struct sockaddr_in에는 AF_INET을 쓰고, socket() 호출에는 PF_INET을 쓰는 것입니다.)

아무튼 그 얘기는 충분합니다. 여러분이 실제로 하고 싶은 일은 getaddrinfo() 호출 결과에서 값을 꺼내 곧바로 socket()에 넣는 것입니다. 이렇게요:

int s;
struct addrinfo hints, *res;

// do the lookup
// [pretend we already filled out the "hints" struct]
getaddrinfo("www.example.com", "http", &hints, &res);

// again, you should do error-checking on getaddrinfo(), and walk
// the "res" linked list looking for valid entries instead of just
// assuming the first one is good (like many of these examples do).
// See the section on client/server for real examples.

s = socket(res->ai_family, res->ai_socktype, res->ai_protocol);

socket()은 이후의 시스템 호출에서 사용할 수 있는 소켓 디스크립터를 반환하고, 오류가 나면 -1을 반환합니다. 전역 변수 errno에는 오류 값이 설정됩니다(자세한 내용과 멀티스레드 프로그램에서 errno를 쓰는 간단한 메모는 errno man 페이지를 보세요).

좋아요, 좋아요. 그런데 이 소켓이 대체 무슨 쓸모가 있냐고요? 답은, 혼자서는 사실 별 쓸모가 없고, 의미 있으려면 계속 읽으며 시스템 호출을 더 해야 한다는 것입니다.

5.3 bind()—나는 어떤 포트에 있나?

소켓을 하나 만들고 나면, 그 소켓을 로컬 머신의 어떤 포트와 연결해야 할 수도 있습니다. (보통 특정 포트에서 들어오는 연결을 listen()하려 할 때 이렇게 합니다. 멀티플레이어 네트워크 게임이 "192.168.5.10의 3490번 포트로 접속하라"고 말하는 것이 이런 경우죠.) 포트 번호는 커널이 들어오는 패킷을 특정 프로세스의 소켓 디스크립터와 대응시키는 데 사용됩니다. 만약 여러분이 connect()만 할 거라면(즉 서버가 아니라 클라이언트라면) 아마 필요 없을 겁니다. 그래도 재미 삼아 읽어 보세요.

아래는 bind() 시스템 호출의 원형입니다:

#include <sys/types.h>
#include <sys/socket.h>

int bind(int sockfd, struct sockaddr *my_addr, int addrlen);

sockfd는 socket()이 반환한 소켓 파일 디스크립터입니다. my_addr는 여러분의 주소, 즉 포트와 IP 주소 정보를 담은 struct sockaddr를 가리키는 포인터입니다. addrlen은 그 주소의 바이트 길이입니다.

휴. 한 번에 받아들이기엔 조금 많네요. 그럼 프로그램이 실행 중인 호스트의 3490번 포트에 소켓을 바인드하는 예제를 봅시다:

struct addrinfo hints, *res;
int sockfd;

// first, load up address structs with getaddrinfo():

memset(&hints, 0, sizeof hints);
hints.ai_family = AF_UNSPEC;  // use IPv4 or IPv6, whichever
hints.ai_socktype = SOCK_STREAM;
hints.ai_flags = AI_PASSIVE;     // fill in my IP for me

getaddrinfo(NULL, "3490", &hints, &res);

// make a socket:

sockfd = socket(res->ai_family, res->ai_socktype, res->ai_protocol);

// bind it to the port we passed in to getaddrinfo():

bind(sockfd, res->ai_addr, res->ai_addrlen);

AI_PASSIVE 플래그를 사용함으로써, 프로그램이 실행 중인 호스트의 IP에 바인드하라고 지시하는 셈입니다. 특정 로컬 IP 주소에 바인드하고 싶다면 AI_PASSIVE를 빼고 getaddrinfo()의 첫 번째 인자에 IP 주소를 넣으면 됩니다.

bind() 역시 오류 시 -1을 반환하고, errno에 오류 값을 설정합니다.

오래된 코드 중 상당수는 bind()를 호출하기 전에 struct sockaddr_in을 손으로 채워 넣습니다. 당연히 이건 IPv4 전용이지만, 일반적으로 getaddrinfo()를 쓰는 편이 더 쉬울 뿐, IPv6에도 같은 식으로 못 할 이유는 없습니다. 어쨌든 옛날 방식 코드는 대략 이렇게 생겼습니다:

// !!! THIS IS THE OLD WAY !!!

int sockfd;
struct sockaddr_in my_addr;

sockfd = socket(PF_INET, SOCK_STREAM, 0);

my_addr.sin_family = AF_INET;
my_addr.sin_port = htons(MYPORT);     // short, network byte order
my_addr.sin_addr.s_addr = inet_addr("10.12.110.57");
memset(my_addr.sin_zero, '\0', sizeof my_addr.sin_zero);

bind(sockfd, (struct sockaddr *)&my_addr, sizeof my_addr);

위 코드에서는 로컬 IP 주소에 바인드하고 싶을 때(위의 AI_PASSIVE 플래그처럼) s_addr 필드에 INADDR_ANY를 넣을 수도 있습니다. IPv6 버전의 INADDR_ANY는 전역 변수 in6addr_any이며, 이것을 struct sockaddr_in6의 sin6_addr 필드에 대입합니다. (변수 초기화에 쓸 수 있는 IN6ADDR_ANY_INIT 매크로도 있습니다.)

bind()를 호출할 때 또 하나 조심할 점은 포트 번호를 너무 낮게 잡지 않는 것입니다. 1024 미만의 모든 포트는 예약돼 있습니다(슈퍼유저가 아니라면요)! 그보다 높은 포트는 65535까지 어떤 것이든 사용할 수 있습니다. 물론 다른 프로그램이 이미 쓰고 있지 않아야 하죠.

가끔 서버를 다시 실행하려고 하면 bind()가 "Address already in use."라고 하며 실패하는 걸 볼 수 있습니다. 이게 무슨 뜻일까요? 이전에 연결되어 있던 소켓의 일부가 아직 커널 안에 남아 있어서 포트를 붙잡고 있는 것입니다. 그냥 조금(1분 정도) 기다려도 되고, 아래처럼 포트를 재사용할 수 있게 코드를 추가해도 됩니다:

int yes=1;
//char yes='1'; // Solaris people use this

// lose the pesky "Address already in use" error message
setsockopt(listener, SOL_SOCKET, SO_REUSEADDR, &yes, sizeof yes);

bind()에 대한 작지만 마지막인 추가 메모 하나 더 있습니다. 절대적으로 bind()를 호출하지 않아도 되는 경우가 있습니다. 원격 머신에 connect()할 때 로컬 포트가 무엇인지는 신경 쓰지 않는다면(telnet처럼 원격 포트만 중요할 때가 그렇죠), 그냥 connect()를 호출하면 됩니다. 그러면 소켓이 아직 바인드되지 않았는지 확인하고, 필요하다면 사용하지 않는 로컬 포트에 자동으로 bind()해 줍니다.

5.4 connect()—거기, 너!

몇 분 동안 여러분이 텔넷 애플리케이션이라고 가정해 봅시다. 사용자가(영화 TRON에서처럼) 소켓 파일 디스크립터를 얻으라고 명령합니다. 여러분은 순순히 socket()을 호출하죠. 다음으로 사용자는 "10.12.110.57"의 "23"번 포트(표준 텔넷 포트)에 연결하라고 말합니다. 이런! 이제 어떻게 하죠?

운이 좋군요, 프로그램 여러분. 지금 바로 원격 호스트에 연결하는 방법을 설명하는 connect() 절을 읽고 있으니까요. 어서 계속 읽으세요! 시간이 없습니다!

connect() 호출은 다음과 같습니다:

#include <sys/types.h>
#include <sys/socket.h>

int connect(int sockfd, struct sockaddr *serv_addr, int addrlen);

sockfd는 socket() 호출이 반환한 우리 동네의 친근한 소켓 파일 디스크립터이고, serv_addr는 목적지 포트와 IP 주소를 담은 struct sockaddr입니다. addrlen은 서버 주소 구조체의 바이트 길이입니다.

이 모든 정보는 멋지게도 getaddrinfo() 호출 결과에서 얻을 수 있습니다.

이제 조금 감이 오기 시작하나요? 여기서는 여러분 목소리가 안 들리니, 그렇다고 믿어 보겠습니다. 그럼 "www.example.com"의 3490번 포트에 소켓 연결을 만드는 예제를 봅시다:

struct addrinfo hints, *res;
int sockfd;

// first, load up address structs with getaddrinfo():

memset(&hints, 0, sizeof hints);
hints.ai_family = AF_UNSPEC;
hints.ai_socktype = SOCK_STREAM;

getaddrinfo("www.example.com", "3490", &hints, &res);

// make a socket:

sockfd = socket(res->ai_family, res->ai_socktype, res->ai_protocol);

// connect!

connect(sockfd, res->ai_addr, res->ai_addrlen);

다시 말하지만, 옛날식 프로그램은 connect()에 넘길 struct sockaddr_in을 직접 채워 넣었습니다. 원하면 그렇게 해도 됩니다. 위의 bind() 절에 있는 비슷한 메모를 참고하세요.

connect()의 반환값은 꼭 확인하세요. 오류가 나면 -1을 반환하고 변수 errno를 설정합니다.

또 하나 눈여겨볼 점은 우리가 bind()를 호출하지 않았다는 것입니다. 기본적으로 우리는 로컬 포트 번호에는 관심이 없고, 어디로 가는지만(원격 포트만) 중요합니다. 커널이 로컬 포트를 알아서 골라 줄 것이고, 연결하는 사이트는 이 정보를 자동으로 받게 됩니다. 걱정할 필요 없습니다.

5.5 listen()—누가 좀 호출해 주세요!

좋아요, 분위기를 좀 바꿔 봅시다. 원격 호스트에 연결하고 싶은 게 아니라면 어떨까요? 그냥 재미 삼아 들어오는 연결을 기다렸다가 뭔가 처리하고 싶다고 해 봅시다. 과정은 두 단계입니다. 먼저 listen()하고, 그다음 accept()합니다(아래 참고).

listen() 호출은 꽤 단순하지만 약간의 설명이 필요합니다:

int listen(int sockfd, int backlog);

sockfd는 늘 보던 socket() 시스템 호출의 소켓 파일 디스크립터입니다. backlog는 들어오는 큐에 허용할 연결 수입니다. 이게 무슨 뜻일까요? 들어오는 연결은 여러분이 accept()할 때까지(아래 참고) 이 큐에서 기다리게 되며, 그때 쌓일 수 있는 최대 개수가 바로 이것입니다. 대부분의 시스템은 이 숫자를 조용히 20 정도로 제한하니, 5나 10 정도로 설정해도 대체로 괜찮을 겁니다.

다시, 늘 그렇듯이 listen()도 오류가 나면 -1을 반환하고 errno를 설정합니다.

아마 짐작하셨겠지만, 서버를 특정 포트에서 돌리려면 listen() 전에 bind()를 호출해야 합니다. (친구들에게 어느 포트로 접속해야 하는지 알려 줄 수 있어야 하니까요!) 그래서 들어오는 연결을 기다리려면 시스템 호출 순서는 다음과 같습니다:

getaddrinfo();
socket();
bind();
listen();
/* accept() goes here */

이건 워낙 자명해서 샘플 코드 대신 그냥 이렇게 두겠습니다. (아래 accept() 절의 코드는 더 완전합니다.) 이 모든 난장판에서 진짜 까다로운 부분은 accept() 호출입니다.

5.6 accept()—“포트 3490으로 전화 주셔서 감사합니다.”

준비하세요. accept() 호출은 좀 이상합니다! 벌어지는 일은 이렇습니다. 아주 멀리 있는 누군가가 여러분이 listen() 중인 포트로 여러분 머신에 connect()하려고 합니다. 그러면 그 연결은 accept()되기를 기다리며 큐에 들어갑니다. 여러분이 accept()를 호출해서 대기 중인 연결을 가져오라고 하면, 그 단일 연결에 사용할 완전히 새로운 소켓 파일 디스크립터를 반환합니다! 맞아요, 갑자기 하나 값으로 소켓 파일 디스크립터 두 개가 생기는 겁니다! 원래 있던 것은 계속 새 연결을 받기 위해 listen()하고 있고, 새로 만들어진 것은 드디어 send()와 recv()를 할 준비가 된 상태입니다. 거의 다 왔습니다!

호출 형태는 다음과 같습니다:

#include <sys/types.h>
#include <sys/socket.h>

int accept(int sockfd, struct sockaddr *addr, socklen_t *addrlen);

sockfd는 listen() 중인 소켓 디스크립터입니다. 쉽죠. addr는 보통 로컬 struct sockaddr_storage에 대한 포인터입니다. 들어오는 연결의 정보가 이곳에 들어갑니다(이걸로 어떤 호스트가 어느 포트에서 여러분을 호출했는지 알아낼 수 있죠). addrlen은 로컬 정수 변수이며, 그 주소를 accept()에 넘기기 전에 sizeof(struct sockaddr_storage)로 설정해 두어야 합니다. accept()는 addr에 그보다 많은 바이트를 쓰지 않습니다. 더 적게 썼다면, 실제 길이를 반영하도록 addrlen 값을 바꿉니다.

놀랍게도 accept()도 오류가 발생하면 -1을 반환하고 errno를 설정합니다. 예상 못 하셨죠?

앞과 마찬가지로 한 번에 소화하기엔 좀 많으니, 아래에 예제 코드 조각을 준비했습니다:

#include <string.h>
#include <sys/types.h>
#include <sys/socket.h>
#include <netdb.h>

#define MYPORT "3490"  // the port users will be connecting to
#define BACKLOG 10     // how many pending connections queue holds

int main(void)
{
    struct sockaddr_storage their_addr;
    socklen_t addr_size;
    struct addrinfo hints, *res;
    int sockfd, new_fd;

    // !! don't forget your error checking for these calls !!

    // first, load up address structs with getaddrinfo():

    memset(&hints, 0, sizeof hints);
    hints.ai_family = AF_UNSPEC;  // use IPv4 or IPv6, whichever
    hints.ai_socktype = SOCK_STREAM;
    hints.ai_flags = AI_PASSIVE;     // fill in my IP for me

    getaddrinfo(NULL, MYPORT, &hints, &res);

    // make a socket, bind it, and listen on it:

    sockfd = socket(res->ai_family, res->ai_socktype,
                                                 res->ai_protocol);
    bind(sockfd, res->ai_addr, res->ai_addrlen);
    listen(sockfd, BACKLOG);

    // now accept an incoming connection:

    addr_size = sizeof their_addr;
    new_fd = accept(sockfd, (struct sockaddr *)&their_addr,
                                                       &addr_size);

    // ready to communicate on socket descriptor new_fd!
    .
    .
    .

다시 한 번 강조하지만, 모든 send()와 recv() 호출에는 소켓 디스크립터 new_fd를 사용하게 됩니다. 만약 딱 한 번의 연결만 받을 거라면, 원한다면 같은 포트로 더 이상 연결이 들어오지 않도록 리스닝 소켓 sockfd를 close()해도 됩니다.

5.7 send()와 recv()—내게 말을 걸어 봐!

이 두 함수는 스트림 소켓이나 연결된 데이터그램 소켓에서 통신할 때 사용합니다. 일반적인 비연결 데이터그램 소켓을 쓰고 싶다면 아래의 sendto()와 recvfrom() 절을 봐야 합니다.

이건 여러분에게 새로울 수도 있고 아닐 수도 있는데, 이 호출들은 블로킹 호출입니다. 즉 recv()는 받을 데이터가 준비될 때까지 멈춰 있습니다. "근데 도대체 블로킹이 뭐예요?!"라고 묻고 싶을 수 있죠. 프로그램이 바로 그 시스템 호출 지점에서 누군가가 무언가를 보낼 때까지 멈춰 있다는 뜻입니다. (운영체제 쪽 전문 용어로는 이런 "멈춤"을 sleep이라고도 하니, 두 표현을 섞어 쓸 수도 있습니다.) send()도 보내는 데이터가 어딘가에서 꽉 막히면 블록될 수 있지만, 그건 좀 더 드문 경우입니다. 이 개념은 나중에 다시 다루면서, 필요할 때 이를 피하는 방법도 이야기하겠습니다.

send() 호출은 다음과 같습니다:

int send(int sockfd, const void *msg, int len, int flags);

sockfd는 데이터를 보내고 싶은 대상 소켓 디스크립터입니다(socket()이 반환한 것이든, accept()로 얻은 것이든 상관없습니다). msg는 보내고 싶은 데이터에 대한 포인터이고, len은 그 데이터의 바이트 길이입니다. flags는 그냥 0으로 두면 됩니다. (플래그에 대한 더 자세한 내용은 send() man 페이지를 보세요.)

예제 코드는 대략 이렇게 생겼습니다:

char *msg = "Beej was here!";
int len, bytes_sent;
.
.
.
len = strlen(msg);
bytes_sent = send(sockfd, msg, len, 0);
.
.
.

send()는 실제로 전송된 바이트 수를 반환합니다. 그런데 이 값이 여러분이 보내라고 한 바이트 수보다 작을 수도 있습니다! 가끔은 데이터를 한꺼번에 왕창 보내라고 해도 다 처리하지 못하거든요. 보낼 수 있는 만큼만 보내고, 나머지는 나중에 여러분이 다시 보내 줄 거라고 기대합니다. 반환값이 len과 다르면, 문자열의 나머지를 보내는 책임은 여러분에게 있다는 점을 기억하세요. 다행인 점은 패킷이 작다면(1K 미만 정도) 아마도 한 번에 전부 보낼 수 있다는 것입니다. 다시 말해 오류가 나면 -1이 반환되고, errno에 오류 번호가 설정됩니다.

recv() 호출은 여러 면에서 비슷합니다:

int recv(int sockfd, void *buf, int len, int flags);

sockfd는 읽어 올 대상 소켓 디스크립터이고, buf는 정보를 읽어 들일 버퍼, len은 그 버퍼의 최대 길이, flags는 역시 0으로 두면 됩니다. (플래그 정보는 recv() man 페이지를 참고하세요.)

recv()는 실제로 버퍼에 읽어 들인 바이트 수를 반환하며, 오류가 나면 -1을 반환합니다(이때 errno도 적절히 설정됩니다).

잠깐! recv()는 0을 반환할 수도 있습니다. 이건 딱 한 가지 의미뿐입니다. 원격 쪽이 연결을 끊었다는 뜻입니다! 0 반환값은 이런 일이 일어났다는 사실을 recv() 방식으로 알려 주는 것입니다.

자, 쉽죠? 이제 스트림 소켓 위에서 데이터를 주고받을 수 있습니다! 와우! 여러분은 유닉스 네트워크 프로그래머입니다!

5.8 sendto()와 recvfrom()—DGRAM 방식으로 대화하기

"좋아요, 다 좋은데,"라고 여러분이 말하는 소리가 들리는군요. "그럼 비연결 데이터그램 소켓은 어떻게 되는 거죠?" 문제없습니다. 딱 맞는 게 있습니다.

데이터그램 소켓은 원격 호스트와 연결되어 있지 않으니, 패킷을 보내기 전에 어떤 정보가 더 필요할까요? 맞습니다! 목적지 주소죠! 설명은 다음과 같습니다:

int sendto(int sockfd, const void *msg, int len, unsigned int flags,
           const struct sockaddr *to, socklen_t tolen);

보시다시피 이 호출은 기본적으로 send()와 같고, 정보 두 개가 더 추가된 것입니다. to는 목적지 IP 주소와 포트를 담은 struct sockaddr에 대한 포인터입니다(아마도 struct sockaddr_in, struct sockaddr_in6, struct sockaddr_storage 같은 것을 마지막에 캐스팅해서 넘기게 될 겁니다). tolen은 내부적으로 int이며, 그냥 sizeof *to 또는 sizeof(struct sockaddr_storage)로 설정하면 됩니다.

목적지 주소 구조체는 아마 getaddrinfo()에서 얻거나, 아래의 recvfrom()에서 얻거나, 아니면 손으로 채워 넣게 될 겁니다.

send()와 마찬가지로 sendto()도 실제로 전송한 바이트 수를 반환합니다(다시 말하지만 여러분이 보내라고 한 바이트 수보다 적을 수 있습니다!). 오류 시에는 -1을 반환합니다.

recv()와 recvfrom()도 마찬가지로 매우 비슷합니다. recvfrom()의 원형은 다음과 같습니다:

int recvfrom(int sockfd, void *buf, int len, unsigned int flags,
             struct sockaddr *from, int *fromlen);

여기서도 마찬가지로, 이건 recv()에 필드 몇 개가 더 붙은 것뿐입니다. from은 송신한 머신의 IP 주소와 포트가 채워질 로컬 struct sockaddr_storage에 대한 포인터입니다. fromlen은 로컬 int에 대한 포인터이며, sizeof *from 또는 sizeof(struct sockaddr_storage)로 초기화해야 합니다. 함수가 반환되면 fromlen에는 실제로 from에 저장된 주소의 길이가 들어 있습니다.

recvfrom()은 받은 바이트 수를 반환하고, 오류 시에는 -1을 반환합니다(errno도 함께 적절히 설정됩니다).

자, 그럼 질문 하나. 왜 소켓 타입으로 struct sockaddr_storage를 쓸까요? 왜 struct sockaddr_in이 아니죠? 이유는 IPv4나 IPv6 어느 한쪽에 묶이고 싶지 않기 때문입니다. 그래서 둘 중 어느 쪽이든 충분히 담을 수 있다고 아는 범용 struct sockaddr_storage를 씁니다.

(그럼 또 다른 질문이 떠오르죠. 왜 struct sockaddr 자체는 어떤 주소든 담기에 충분히 크지 않을까요? 우리는 범용 struct sockaddr_storage를 또 범용 struct sockaddr로 캐스팅하기까지 하는데요! 쓸데없이 중복돼 보이지 않나요? 답은 그냥 충분히 크지 않기 때문입니다. 그리고 지금 시점에서 바꾸려면 문제가 꽤 클 것 같았겠죠. 그래서 새 걸 만든 겁니다.)

기억하세요. 데이터그램 소켓에 connect()를 해 두면 이후의 모든 통신에 send()와 recv()를 간단히 사용할 수 있습니다. 소켓 자체는 여전히 데이터그램 소켓이고 패킷도 여전히 UDP를 사용하지만, 소켓 인터페이스가 목적지와 출발지 정보를 자동으로 붙여 줍니다.

5.9 close()와 shutdown()—이제 그만 가 줘!

휴! 하루 종일 send()하고 recv()하느라 지치셨군요. 이제 소켓 디스크립터의 연결을 닫을 준비가 됐습니다. 어렵지 않습니다. 그냥 일반적인 유닉스 파일 디스크립터 함수 close()를 쓰면 됩니다:

close(sockfd);

이렇게 하면 그 소켓에 대해 더 이상의 읽기와 쓰기가 불가능해집니다. 원격 쪽에서 그 소켓에 읽기나 쓰기를 시도하면 오류를 받게 됩니다.

소켓이 닫히는 방식을 조금 더 세밀하게 제어하고 싶다면 shutdown() 함수를 사용할 수 있습니다. 이 함수는 특정 방향의 통신만 끊거나, 양방향 모두 끊을 수 있게 해 줍니다(close()처럼요). 원형은 다음과 같습니다:

int shutdown(int sockfd, int how);

sockfd는 종료하고 싶은 소켓 파일 디스크립터이고, how는 다음 값 중 하나입니다:

how 효과
0 이후 수신 금지
1 이후 송신 금지
2 이후 송신과 수신 모두 금지(close()와 비슷함)

shutdown()은 성공 시 0, 오류 시 -1을 반환합니다(이때 errno도 적절히 설정됩니다).

비연결 데이터그램 소켓에 shutdown()을 굳이 사용하더라도, 이후 send()와 recv()를 더 이상 사용할 수 없게 만들 뿐입니다(데이터그램 소켓을 connect()하면 이 함수들을 쓸 수 있다는 점을 기억하세요).

중요한 점은 shutdown()이 실제로 파일 디스크립터를 닫는 것은 아니라는 것입니다. 단지 사용 가능 여부만 바꿉니다. 소켓 디스크립터를 해제하려면 close()를 사용해야 합니다.

이게 전부입니다.

(단, Windows와 Winsock을 사용 중이라면 close() 대신 closesocket()을 호출해야 한다는 것만 기억하세요.)

5.10 getpeername()—당신은 누구?

이 함수는 정말 쉽습니다.

너무 쉬워서 별도 절을 만들지 않을 뻔했어요. 그래도 일단 넣어 두겠습니다.

getpeername() 함수는 연결된 스트림 소켓의 반대편에 누가 있는지 알려 줍니다. 원형은 다음과 같습니다:

#include <sys/socket.h>

int getpeername(int sockfd, struct sockaddr *addr, int *addrlen);

sockfd는 연결된 스트림 소켓의 디스크립터이고, addr는 연결 반대편의 정보를 담을 struct sockaddr(또는 struct sockaddr_in)에 대한 포인터입니다. addrlen은 int에 대한 포인터이며, sizeof *addr 또는 sizeof(struct sockaddr)로 초기화해 두어야 합니다.

이 함수는 오류 시 -1을 반환하고 그에 맞게 errno를 설정합니다.

상대의 주소를 얻고 나면 inet_ntop(), getnameinfo(), 또는 gethostbyaddr()를 사용해 출력하거나 추가 정보를 얻을 수 있습니다. 아니요, 로그인 이름은 얻을 수 없습니다. (좋아요, 좋아요. 상대 컴퓨터가 ident 데몬을 돌리고 있다면 가능하긴 합니다. 하지만 그건 이 문서의 범위를 벗어납니다. 자세한 내용은 RFC 141323을 참고하세요.)

5.11 gethostname()—나는 누구?

getpeername()보다 더 쉬운 것이 gethostname() 함수입니다. 이 함수는 여러분 프로그램이 실행 중인 컴퓨터의 이름을 반환합니다. 그리고 그 이름은 위에서 본 getaddrinfo()로 넘겨 로컬 머신의 IP 주소를 알아내는 데 사용할 수 있습니다.

이보다 더 재미있는 게 뭐가 있을까요? 몇 가지 떠오르긴 하지만, 소켓 프로그래밍과는 상관없군요. 어쨌든 형태는 이렇습니다:

#include <unistd.h>

int gethostname(char *hostname, size_t size);

인자는 간단합니다. hostname은 함수가 반환될 때 호스트 이름이 들어갈 문자 배열에 대한 포인터이고, size는 그 hostname 배열의 바이트 길이입니다.

함수는 성공적으로 끝나면 0을 반환하고, 오류가 나면 늘 그렇듯 errno를 설정한 뒤 -1을 반환합니다.
