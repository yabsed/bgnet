8 자주 묻는 질문

저 헤더 파일들은 어디서 구하나요?

시스템에 이미 없다면 아마 필요하지 않은 것일 가능성이 큽니다. 사용 중인 플랫폼의 매뉴얼을 확인하세요. Windows용으로 빌드한다면 #include <winsock.h>만 있으면 됩니다.

bind()가 “Address already in use”를 보고하면 어떻게 해야 하나요?

리스닝 소켓에 SO_REUSEADDR 옵션과 함께 setsockopt()를 사용해야 합니다. 예제는 bind() 절과 select() 절을 참고하세요.

시스템에서 열려 있는 소켓 목록은 어떻게 보나요?

netstat를 사용하세요. 자세한 내용은 man 페이지를 보면 되지만, 일단 다음처럼 입력해도 꽤 괜찮은 출력이 나옵니다.

$ netstat

유일한 문제는 어떤 소켓이 어떤 프로그램과 연결되어 있는지 알아내는 일입니다. :-)

라우팅 테이블은 어떻게 보나요?

route 명령(대부분의 Linux에서는 /sbin 아래에 있습니다), netstat -r, 또는 ip route를 실행하면 됩니다.

컴퓨터가 한 대뿐이면 클라이언트와 서버 프로그램은 어떻게 실행하나요? 네트워크 프로그램을 작성하려면 네트워크가 꼭 필요한 것 아닌가요?

다행히 거의 모든 머신은 커널 안에서 동작하며 네트워크 카드인 척하는 루프백 네트워크 “장치”를 구현하고 있습니다. (라우팅 테이블에 “lo”로 나타나는 인터페이스가 바로 그것입니다.)

여러분이 “goat”라는 이름의 머신에 로그인했다고 가정해 봅시다. 한 창에서는 클라이언트를 실행하고 다른 창에서는 서버를 실행하면 됩니다. 또는 서버를 백그라운드로 띄운 뒤 (“server &”) 같은 창에서 클라이언트를 실행해도 됩니다. 루프백 장치의 핵심은 client goat나 client localhost를 사용할 수 있다는 점입니다. (“localhost”는 보통 /etc/hosts 파일에 정의되어 있습니다.) 즉, 실제 네트워크가 없어도 클라이언트와 서버가 통신할 수 있습니다!

요컨대, 네트워크가 없는 단일 머신에서 실행하기 위해 기존 코드에 어떤 수정도 할 필요가 없습니다. 만세!

원격 측이 연결을 닫았는지는 어떻게 알 수 있나요?

recv()가 0을 반환하면 알 수 있습니다.

“ping” 유틸리티는 어떻게 구현하나요? ICMP는 무엇인가요? raw socket과 SOCK_RAW에 대한 더 자세한 내용은 어디서 볼 수 있나요?

raw socket에 관한 질문은 W. Richard Stevens의 UNIX Network Programming 책에서 답을 찾을 수 있습니다. 또한 Stevens의 UNIX Network Programming 소스 코드 안의 ping/ 하위 디렉터리도 보세요. 온라인에서 구할 수 있습니다44.

connect() 호출의 타임아웃을 변경하거나 줄이려면 어떻게 하나요?

W. Richard Stevens가 해 줄 답을 그대로 반복하는 대신, UNIX Network Programming 소스 코드의 lib/connect_nonb.c45를 참고하라고 말씀드리겠습니다.

요점은 이렇습니다. 먼저 socket()으로 소켓 디스크립터를 만들고, 논블로킹으로 설정한 뒤 connect()를 호출합니다. 모든 것이 잘 되면 connect()는 즉시 -1을 반환하고 errno는 EINPROGRESS로 설정됩니다. 그다음 원하는 타임아웃을 넣어 select()를 호출하고, 그 소켓 디스크립터를 읽기 집합과 쓰기 집합 양쪽 모두에 넣습니다. 타임아웃되지 않았다면 connect() 호출이 완료되었다는 뜻입니다. 이 시점에서는 getsockopt()와 SO_ERROR 옵션을 사용해서 connect() 호출의 반환 값을 확인해야 하며, 오류가 없었다면 그 값은 0이어야 합니다.

마지막으로, 실제 데이터 전송을 시작하기 전에 소켓을 다시 블로킹 모드로 되돌리고 싶을 가능성이 큽니다.

이 방식에는 연결이 진행되는 동안 프로그램이 다른 일을 할 수 있다는 추가 장점도 있습니다. 예를 들어 타임아웃을 500ms처럼 짧게 두고, 타임아웃될 때마다 화면의 상태 표시를 갱신한 뒤 다시 select()를 호출할 수 있습니다. 그렇게 20번쯤 타임아웃이 반복되면, 이제 연결을 포기할 때가 되었다고 판단할 수 있겠죠.

말했듯이, 아주 훌륭한 예제가 들어 있으니 Stevens의 소스를 직접 보세요.

우선 Windows를 지우고 Linux나 BSD를 설치하세요. };-) 농담이고, 사실은 소개 부분의 Windows 빌드 절을 보면 됩니다.

Solaris/SunOS용으로 빌드하려면 어떻게 하나요? 컴파일할 때 계속 링커 에러가 납니다!

Sun 계열 시스템은 소켓 라이브러리를 자동으로 링크하지 않기 때문에 그런 링커 에러가 납니다. 어떻게 해야 하는지는 소개 부분의 Solaris/SunOS 빌드 절 예제를 참고하세요.

왜 select()는 시그널이 오면 계속 빠져나오나요?

시그널은 블록된 시스템 호출이 -1을 반환하게 만들고, errno를 EINTR로 설정하는 경향이 있습니다. sigaction()으로 시그널 핸들러를 설정할 때 SA_RESTART 플래그를 줄 수 있는데, 이 플래그는 시스템 호출이 인터럽트된 뒤 다시 시작되도록 의도된 것입니다.

물론 이것이 항상 잘 동작하는 것은 아닙니다.

제가 가장 좋아하는 해결책은 goto를 쓰는 것입니다. 교수님들이 이런 걸 정말 싫어한다는 걸 알고 있죠? 그러니 과감히 써 보세요!

select_restart:
if ((err = select(fdmax+1, &readfds, NULL, NULL, NULL)) == -1) {
    if (errno == EINTR) {
        // some signal just interrupted us, so restart
        goto select_restart;
    }
    // handle the real error here:
    perror("select");
}

물론 이 경우 꼭 goto를 써야 하는 것은 아닙니다. 다른 제어 구조를 써도 됩니다. 하지만 저는 이 상황에서는 goto가 오히려 더 깔끔하다고 생각합니다.

recv() 호출에 타임아웃을 구현하려면 어떻게 하나요?

select()를 사용하세요! 읽으려는 소켓 디스크립터에 대해 타임아웃 값을 지정할 수 있습니다. 또는 전체 기능을 아래처럼 하나의 함수로 감쌀 수도 있습니다.

#include <unistd.h>
#include <sys/time.h>
#include <sys/types.h>
#include <sys/socket.h>

int recvtimeout(int s, char *buf, int len, int timeout)
{
    fd_set fds;
    int n;
    struct timeval tv;

    // set up the file descriptor set
    FD_ZERO(&fds);
    FD_SET(s, &fds);

    // set up the struct timeval for the timeout
    tv.tv_sec = timeout;
    tv.tv_usec = 0;

    // wait until timeout or data received
    n = select(s+1, &fds, NULL, NULL, &tv);
    if (n == 0) return -2; // timeout!
    if (n == -1) return -1; // error

    // data must be here, so do a normal recv()
    return recv(s, buf, len, 0);
}
.
.
.
// Sample call to recvtimeout():
n = recvtimeout(s, buf, sizeof buf, 10); // 10 second timeout

if (n == -1) {
    // error occurred
    perror("recvtimeout");
}
else if (n == -2) {
    // timeout occurred
} else {
    // got some data in buf
}
.
.
.

recvtimeout()는 타임아웃이 발생하면 -2를 반환한다는 점을 보세요. 왜 0을 반환하지 않을까요? 기억하시겠지만, recv()가 0을 반환하는 것은 원격 측이 연결을 닫았다는 의미입니다. 그러니 그 반환값은 이미 다른 뜻으로 사용 중이고, -1은 “오류”를 뜻하므로, 저는 타임아웃 표시로 -2를 택했습니다.

소켓으로 보내기 전에 데이터를 암호화하거나 압축하려면 어떻게 하나요?

암호화를 위한 쉬운 방법 하나는 SSL(secure sockets layer)을 쓰는 것이지만, 그것은 이 가이드의 범위를 벗어납니다. (자세한 내용은 OpenSSL 프로젝트46를 참고하세요.)

하지만 여러분이 직접 압축기나 암호화 시스템을 끼워 넣거나 구현하고 싶다면, 데이터를 양 끝단 사이에서 여러 단계를 거쳐 흐르는 것으로 생각하면 됩니다. 각 단계는 데이터를 어떤 식으로든 바꿉니다.

  1. 서버가 파일(또는 다른 곳)에서 데이터를 읽는다
  2. 서버가 데이터를 암호화/압축한다 (이 부분은 여러분이 추가한다)
  3. 서버가 암호화된 데이터를 send()한다

반대로 수신 측은 이렇습니다.

  1. 클라이언트가 암호화된 데이터를 recv()한다
  2. 클라이언트가 데이터를 복호화/압축 해제한다 (이 부분은 여러분이 추가한다)
  3. 클라이언트가 데이터를 파일(또는 다른 곳)에 쓴다

압축과 암호화를 함께 할 거라면, 압축을 먼저 한다는 것만 기억하세요. :-)

클라이언트가 서버가 한 일을 제대로 되돌리기만 하면, 중간 단계가 몇 개가 더 들어가든 최종적으로 데이터는 문제없이 복원됩니다.

그러니 제 코드를 이용하려면, 데이터가 읽힌 뒤 네트워크로 send()되기 전 사이 지점을 찾아 그곳에 암호화 코드를 끼워 넣기만 하면 됩니다.

자꾸 보이는 “PF_INET”은 무엇인가요? AF_INET과 관련이 있나요?

네, 맞습니다. 자세한 내용은 socket() 절을 보세요.

클라이언트가 보낸 셸 명령을 받아 실행하는 서버는 어떻게 작성하나요?

단순화를 위해 클라이언트가 connect()한 뒤, send()하고, 곧바로 연결을 close()한다고 가정합시다. (즉, 다시 연결하기 전에는 추가 시스템 호출이 없는 상황입니다.)

클라이언트 쪽 절차는 이렇습니다.

  1. 서버에 connect()한다
  2. send("/sbin/ls > /tmp/client.out")
  3. 연결을 close()한다

한편 서버는 데이터를 받아 처리하고 실행합니다.

  1. 클라이언트의 연결을 accept()한다
  2. 명령 문자열을 recv(str)로 받는다
  3. 연결을 close()한다
  4. system(str)로 명령을 실행한다

주의하세요! 서버가 클라이언트가 시키는 대로 실행하게 두는 것은 원격 셸 접근 권한을 주는 것과 비슷해서, 누군가 서버에 연결했을 때 여러분 계정에 대해 온갖 일을 할 수 있습니다. 예를 들어 위 상황에서 클라이언트가 “rm -rf ~”를 보내면 어떻게 될까요? 여러분 계정의 모든 것이 지워집니다.

그래서 정신을 차리고, foobar 같은 안전하다고 확신하는 몇몇 유틸리티 외에는 클라이언트가 사용할 수 없도록 막는다고 해 봅시다.

if (!strncmp(str, "foobar", 6)) {
    sprintf(sysstr, "%s > /tmp/server.out", str);
    system(sysstr);
}

하지만 안타깝게도 아직도 안전하지 않습니다. 클라이언트가 “foobar; rm -rf ~”를 입력하면 어떻게 될까요? 가장 안전한 방법은 명령 인수 안의 모든 영숫자가 아닌 문자 앞에(상황에 따라 공백도 포함해서) 이스케이프 문자 “\”를 붙여 주는 작은 루틴을 작성하는 것입니다.

보시다시피, 서버가 클라이언트가 보낸 것을 실행하기 시작하면 보안은 매우 큰 문제가 됩니다.

많은 데이터를 보내고 있는데 recv()를 하면 한 번에 536바이트나 1460바이트씩만 받습니다. 그런데 로컬 머신에서 실행하면 데이터를 한꺼번에 다 받습니다. 왜 그런가요?

여러분은 MTU, 즉 물리 매체가 처리할 수 있는 최대 크기에 걸리고 있는 것입니다. 로컬 머신에서는 루프백 장치를 쓰므로 8K 이상도 문제없이 다룰 수 있습니다. 하지만 Ethernet은 헤더를 포함해도 1500바이트까지만 처리할 수 있으니 그 한계에 걸립니다. 모뎀 환경에서는 MTU가 576이므로 (마찬가지로 헤더 포함) 더 낮은 한계에 걸립니다.

우선 모든 데이터가 실제로 전송되고 있는지부터 확인해야 합니다. (자세한 내용은 sendall() 함수 구현을 보세요.) 그게 확인되면, 모든 데이터가 읽힐 때까지 recv()를 반복 호출해야 합니다.

여러 번의 recv() 호출로 완전한 패킷을 받는 방법은 데이터 캡슐화의 후속편 절을 참고하세요.

Windows 환경인데 fork() 시스템 호출도 없고 struct sigaction 같은 것도 없습니다. 어떻게 해야 하나요?

만약 있다면, 아마 여러분 컴파일러와 함께 제공된 POSIX 라이브러리 안에 있을 것입니다. 제가 Windows 머신이 없어서 정확한 답은 드리기 어렵지만, Microsoft에 POSIX 호환 계층이 있었고 그 안에 fork()가 있었던 것으로 기억합니다. (sigaction도 어쩌면 있을 수 있습니다.)

VC++에 포함된 도움말에서 “fork”나 “POSIX”를 검색해서 단서를 찾아보세요.

그마저도 전혀 안 된다면 fork()/sigaction 관련 부분은 버리고, Win32 쪽 대응물인 CreateProcess()로 바꾸세요. 저는 CreateProcess()를 어떻게 쓰는지 모릅니다. 인자가 엄청나게 많거든요. 하지만 VC++ 문서에 설명이 있을 것입니다.

방화벽 뒤에 있는데, 외부 사람이 제 IP 주소를 알고 제 머신에 접속할 수 있게 하려면 어떻게 하나요?

안타깝지만 방화벽의 목적 자체가 외부 사람이 방화벽 내부 머신에 접속하는 것을 막는 것이므로, 그런 접속을 허용하는 행위는 기본적으로 보안 위반으로 간주됩니다.

그렇다고 모든 방법이 막힌 것은 아닙니다. 예를 들어 방화벽이 어떤 형태의 masquerading이나 NAT 같은 것을 하고 있다면, 여전히 바깥으로 connect()하는 경우가 많습니다. 항상 여러분 쪽에서 연결을 먼저 시작하도록 프로그램을 설계하면 괜찮을 것입니다.

그것으로 만족할 수 없다면, 시스템 관리자에게 방화벽에 구멍을 하나 내서 외부 사람이 여러분에게 접속할 수 있도록 요청할 수 있습니다. 방화벽은 NAT 소프트웨어를 통해서든, 프록시 같은 것을 통해서든 여러분에게 트래픽을 전달할 수 있습니다.

다만 방화벽에 구멍을 뚫는 일은 가볍게 볼 일이 아닙니다. 악의적인 사람이 내부 네트워크에 접근하지 못하도록 확실히 해야 합니다. 초보자라면 소프트웨어를 안전하게 만드는 일이 생각보다 훨씬 어렵다는 점을 꼭 알아 두세요.

시스템 관리자가 저 때문에 화내게 만들진 마세요. ;-)

패킷 스니퍼는 어떻게 만들죠? 이더넷 인터페이스를 promiscuous mode로 두려면 어떻게 하나요?

잘 모르시는 분들을 위해 설명하자면, 네트워크 카드가 “promiscuous mode”에 있으면 이 특정 머신 앞으로 온 패킷만이 아니라 모든 패킷을 운영체제로 넘기게 됩니다. (여기서 말하는 것은 IP 주소가 아니라 Ethernet 계층 주소입니다. 다만 Ethernet이 IP보다 하위 계층이므로 결과적으로 모든 IP 패킷도 함께 전달됩니다. 더 자세한 내용은 하위 수준의 난센스와 네트워크 이론 절을 보세요.)

이것이 바로 패킷 스니퍼가 동작하는 기본 원리입니다. 인터페이스를 promiscuous mode로 바꾸면, 운영체제가 선로를 지나가는 모든 패킷을 받게 됩니다. 그러면 여러분은 그 데이터를 읽을 수 있는 어떤 종류의 소켓을 하나 가지게 됩니다.

안타깝게도 이 질문의 답은 플랫폼에 따라 달라집니다. 하지만 예를 들어 “windows promiscuous ioctl” 같은 키워드로 검색해 보면 뭔가 실마리를 찾을 가능성이 큽니다. Linux 쪽으로는 유용해 보이는 Stack Overflow 스레드47도 있습니다.

TCP나 UDP 소켓에 사용자 지정 타임아웃 값을 설정하려면 어떻게 하나요?

시스템에 따라 다릅니다. 여러분 시스템이 그런 기능을 지원하는지 보려면 setsockopt()와 함께 사용하는 SO_RCVTIMEO, SO_SNDTIMEO를 검색해 보세요.

Linux 매뉴얼 페이지에서는 대안으로 alarm()이나 setitimer()를 사용하라고 제안합니다.

사용 가능한 포트는 어떻게 알 수 있나요? “공식” 포트 번호 목록이 있나요?

보통은 큰 문제가 되지 않습니다. 예를 들어 웹 서버를 작성한다면 잘 알려진 포트 80을 쓰는 것이 좋습니다. 그냥 여러분만의 특수한 서버를 만들고 있다면, 임의의 포트를 하나 골라(단, 1023보다 큰 값으로) 시도해 보면 됩니다.

이미 사용 중인 포트라면 bind()를 시도할 때 “Address already in use” 오류가 날 것입니다. 다른 포트를 고르세요. (설정 파일이나 명령행 옵션으로 사용자가 대체 포트를 지정할 수 있게 해 두는 것도 좋은 생각입니다.)

Internet Assigned Numbers Authority(IANA)가 관리하는 공식 포트 번호 목록48도 있습니다. 다만 그 목록에 어떤 포트(1023 초과)가 있다고 해서 여러분이 그 포트를 못 쓰는 것은 아닙니다. 예를 들어 Id Software의 DOOM은 “mdqs”와 같은 포트를 사용합니다. 그게 뭔지는 모르겠지만요. 중요한 것은 여러분이 그 포트를 쓰고 싶을 때, 같은 머신에서 다른 누군가가 그 포트를 사용하고 있지 않아야 한다는 점뿐입니다.
