3 IP 주소, 구조체, 그리고 데이터 가공

드디어 코드 이야기를 좀 할 차례가 왔습니다.

하지만 먼저 코드가 아닌 이야기부터 조금 더 해 봅시다. 좋죠! 우선 IP 주소와 포트에 대해 잠깐 짚고 넘어가겠습니다. 그래야 뒤에서 소켓 API가 IP 주소와 다른 데이터를 어떻게 저장하고 다루는지 이야기할 수 있습니다.

3.1 IP 주소, 버전 4와 6

벤 케노비가 아직 오비완 케노비라고 불리던 옛날 옛적에는, 인터넷 프로토콜 버전 4, 즉 IPv4라고도 부르는 멋진 네트워크 라우팅 시스템이 있었습니다. 주소는 네 바이트(즉 네 개의 "옥텟")로 이루어졌고, 보통 192.0.2.111처럼 점으로 구분된 숫자 형태로 표기했습니다.

아마 한 번쯤은 본 적이 있을 겁니다.

실제로 이 글을 쓰는 시점에도 인터넷의 거의 모든 사이트가 IPv4를 사용합니다.

오비완을 포함한 모두는 행복했습니다. 그러다 Vint Cerf라는 어떤 비관론자가 우리가 곧 IPv4 주소를 다 써 버릴 거라고 경고하기 전까지는 말이죠!

(Vint Cerf는 암울한 IPv4 대재앙의 도래를 경고했을 뿐 아니라, 인터넷의 아버지14로도 잘 알려져 있습니다. 그러니 제가 그의 판단을 함부로 의심할 입장은 아니죠.)

주소를 다 쓴다고요? 어떻게 그럴 수 있죠? 32비트 IPv4 주소 안에는 수십억 개의 IP 주소가 있잖아요. 정말 세상에 컴퓨터가 수십억 대나 있는 걸까요?

네, 그렇습니다.

게다가 초창기에는 컴퓨터가 몇 대 없었고, 모두가 10억이라는 숫자를 상상도 못 할 만큼 큰 값이라고 여겼습니다. 그래서 몇몇 대형 조직은 자신들만 쓰라고 수백만 개의 IP 주소를 넉넉히 배정받았습니다. (예를 들면 Xerox, MIT, Ford, HP, IBM, GE, AT&T, 그리고 Apple 같은 회사들이죠.)

사실 여러 임시방편이 없었다면 우리는 훨씬 전에 주소를 다 써 버렸을 겁니다.

하지만 이제는 모든 사람, 모든 컴퓨터, 모든 계산기, 모든 전화기, 모든 주차 요금계, 심지어 왜 안 되겠냐는 듯 모든 강아지까지도 IP 주소를 갖는 시대에 살고 있습니다.

그래서 IPv6가 탄생했습니다. Vint Cerf는 아마 불사신일 테고(설령 그의 육체가 사라진다 해도, Internet2 어딘가 깊숙한 곳에서 초지능형 ELIZA15 프로그램 형태로 이미 존재하고 있을지도 모르죠), 다음 버전의 인터넷 프로토콜에서도 주소가 부족해져서 그가 다시 "내가 그랬잖아"라고 말하는 꼴은 아무도 보고 싶지 않았습니다.

이 말이 무엇을 뜻할까요?

주소가 훨씬 더 많이 필요하다는 뜻입니다. 두 배도 아니고, 10억 배도 아니고, 천조 배도 아니라, 무려 79 옥틸리언 배나 더 많은 가능한 주소 수가 필요하다는 말이죠! 이쯤 되면 충분하겠죠!

"Beej, 그거 진짜예요? 큰 수는 도무지 못 믿겠는데요."라고 말하고 싶을 수도 있습니다. 하지만 32비트와 128비트의 차이가 겉보기엔 그리 크지 않아 보여도, 여기서는 거듭제곱이 중요합니다. 32비트는 약 40억 개의 수(232)를 나타내고, 128비트는 약 340조의 조의 조 개의 수(정말로, 2128)를 나타냅니다. 이건 우주의 모든 별 하나마다 IPv4 인터넷 백만 개를 줄 수 있는 수준입니다.

IPv4의 점-숫자 형식은 이제 잊으세요. 이제는 두 바이트씩 끊어서 콜론으로 구분한 16진수 표기법을 씁니다. 예를 들면 이렇게요:

2001:0db8:c9d2:aee5:73e3:934a:a5ae:9551

그게 다가 아닙니다! IP 주소에 0이 많이 들어 있을 때는 두 개의 콜론 사이로 압축할 수 있고, 각 바이트 쌍의 앞쪽 0도 생략할 수 있습니다. 예를 들어 아래 주소 쌍들은 서로 같습니다:

2001:0db8:c9d2:0012:0000:0000:0000:0051
2001:db8:c9d2:12::51

2001:0db8:ab00:0000:0000:0000:0000:0000
2001:db8:ab00::

0000:0000:0000:0000:0000:0000:0000:0001
::1

::1 주소는 루프백 주소입니다. 언제나 "지금 이 프로그램이 실행 중인 이 머신"을 뜻합니다. IPv4의 루프백 주소는 127.0.0.1입니다.

마지막으로, 가끔 IPv6 주소에서 볼 수 있는 IPv4 호환 모드가 있습니다. 예를 들어 IPv4 주소 192.0.2.33을 IPv6 주소로 표현하고 싶다면 다음 표기를 사용합니다: "::ffff:192.0.2.33".

이제 좀 신나는 이야기죠.

사실 너무 신나서, IPv6를 만든 사람들은 예약 용도로 수조의 수조 개 주소를 아주 태연하게 잘라내 버렸습니다. 그래도 주소가 워낙 많아서 이제는 누가 그 수를 세겠어요? 은하계 모든 행성의 모든 남녀노소와 강아지, 주차 요금계까지 다 쓰고도 남습니다. 그리고 믿으세요, 은하계의 모든 행성에는 주차 요금계가 있습니다. 진짜예요.

3.1.1 서브넷

조직상 편의를 위해, "이 IP 주소의 앞부분부터 여기까지는 네트워크 부분이고 나머지는 호스트 부분이다"라고 구분해 두면 편할 때가 있습니다.

예를 들어 IPv4 주소가 192.0.2.12라면, 앞의 세 바이트는 네트워크이고 마지막 한 바이트는 호스트라고 말할 수 있습니다. 다시 말해 네트워크 192.0.2.0의 호스트 12를 뜻하는 셈입니다(호스트였던 바이트를 0으로 바꾼 것을 보세요).

이제 좀 오래된 정보도 하나 더 보겠습니다. 준비됐나요? 아주 먼 옛날에는 서브넷에 "클래스"가 있어서 주소의 첫 1, 2, 3바이트가 네트워크 부분이었습니다. 네트워크 1바이트와 호스트 3바이트를 가질 만큼 운이 좋았다면 네트워크 안에 24비트 분량의 호스트(약 1600만 대)를 둘 수 있었습니다. 이것이 "Class A" 네트워크였습니다. 반대편 끝에는 네트워크 3바이트와 호스트 1바이트(예약된 몇 개를 제외하면 256호스트)를 가진 "Class C"가 있었습니다.

그래서 보듯이 Class A는 아주 적었고, Class C는 엄청 많았으며, 그 중간쯤에 Class B가 있었습니다.

IP 주소의 네트워크 부분은 넷마스크라는 것으로 설명하며, IP 주소에 대해 비트 단위 AND를 하면 네트워크 번호를 얻을 수 있습니다. 넷마스크는 보통 255.255.255.0처럼 생겼습니다. (예를 들어 넷마스크가 이것이고 IP가 192.0.2.12라면, 네트워크는 192.0.2.12 AND 255.255.255.0이어서 192.0.2.0이 됩니다.)

안타깝게도 이것만으로는 인터넷이 나중에 필요로 하게 될 세밀함을 감당할 수 없었습니다. Class C 네트워크는 빠르게 고갈됐고, Class A는 아예 다 떨어졌으니 말 다 했죠. 그래서 권한 있는 분들이 넷마스크를 8, 16, 24비트에만 제한하지 않고 임의의 비트 수로 지정할 수 있게 했습니다. 예를 들어 넷마스크를 255.255.255.252로 두면 네트워크 30비트, 호스트 2비트가 되어 네트워크에 4개의 호스트를 둘 수 있습니다. (넷마스크는 항상 1비트들이 나오다가 그 뒤로 0비트들이 이어지는 형태라는 점에 주의하세요.)

하지만 255.192.0.0 같은 큰 숫자 문자열을 넷마스크로 쓰는 건 좀 불편합니다. 사람들이 몇 비트인지 직관적으로 떠올리기 어렵고, 표기 자체도 길거든요. 그래서 더 나은 새 방식이 등장했습니다. IP 주소 뒤에 슬래시를 붙이고 네트워크 비트 수를 10진수로 적으면 됩니다. 예: 192.0.2.12/30.

IPv6라면 2001:db8::/32 또는 2001:db8:5413:4028::9db9/64 같은 식입니다.

3.1.2 포트 번호

기억하시겠지만, 앞서 계층형 네트워크 모델을 소개하면서 인터넷 계층(IP)과 호스트-대-호스트 전송 계층(TCP와 UDP)을 나눠서 설명했습니다. 다음 문단을 보기 전에 그 내용을 다시 떠올려 보세요.

알고 보니 IP 계층에서 쓰는 IP 주소 말고도 TCP(스트림 소켓)와 우연히도 UDP(데이터그램 소켓)에서도 쓰는 주소가 하나 더 있습니다. 바로 포트 번호입니다. 16비트 숫자이며 연결의 로컬 주소 같은 역할을 합니다.

IP 주소를 호텔의 도로명 주소로 생각하고, 포트 번호를 객실 번호라고 생각해 보세요. 꽤 괜찮은 비유입니다. 나중에는 자동차 산업을 끌어들인 비유도 떠올릴지 모르겠네요.

한 IP 주소를 가진 컴퓨터가 메일 서비스도 처리하고 웹 서비스도 처리하려면, 둘을 어떻게 구분할까요?

인터넷의 서비스마다 잘 알려진 포트 번호가 따로 있습니다. IANA 전체 포트 목록16이나, 유닉스 계열 시스템이라면 /etc/services 파일에서 모두 볼 수 있습니다. HTTP(웹)는 80번, telnet은 23번, SMTP는 25번, 게임 DOOM17은 666번 포트를 사용했습니다. 1024 미만의 포트는 보통 특별한 것으로 간주되어, 사용하려면 운영체제의 특별 권한이 필요한 경우가 많습니다.

이제 거의 끝입니다!

3.2 바이트 순서

왕국의 명에 따라! 앞으로 바이트 순서는 형편없는 것과 훌륭한 것, 두 종류로 나뉜다!

농담입니다만, 실제로 한쪽이 더 낫긴 합니다. :-)

이건 정말 돌려 말하기 어렵네요. 그냥 말하자면, 여러분의 컴퓨터는 몰래 바이트를 거꾸로 저장하고 있을지도 모릅니다. 네, 말하고 싶지 않았지만요.

문제는 인터넷 세계의 모두가 두 바이트짜리 16진수 b34f를 표현할 때 순서대로 b3 다음 4f로 저장하자는 데 대체로 합의했다는 것입니다. 말이 되죠. 그리고 Wilford Brimley18가 말했을 법하게, 이게 바로 옳은 방식입니다. 이렇게 큰 쪽 바이트를 먼저 저장하는 방식을 빅 엔디언이라고 합니다.

안타깝게도 세상 곳곳의 일부 컴퓨터, 즉 Intel 또는 Intel 호환 프로세서를 쓰는 기계는 바이트를 반대로 저장합니다. 그래서 b34f는 메모리에서 4f 다음 b3 순서로 저장됩니다. 이런 저장 방식을 리틀 엔디언이라고 합니다.

그런데 용어는 여기서 끝이 아닙니다! 좀 더 제정신인 빅 엔디언은 네트워크 쪽 사람들이 좋아하는 순서라서 네트워크 바이트 순서라고도 부릅니다.

여러분의 컴퓨터는 숫자를 호스트 바이트 순서로 저장합니다. Intel 80x86이라면 리틀 엔디언이고, Motorola 68k라면 빅 엔디언입니다. PowerPC라면... 음, 경우에 따라 다릅니다!

패킷을 만들거나 데이터 구조를 채울 때는 2바이트와 4바이트 숫자가 네트워크 바이트 순서인지 확인해야 하는 경우가 많습니다. 그런데 호스트의 고유한 바이트 순서를 모르면 어떻게 하죠?

좋은 소식입니다! 그냥 호스트 바이트 순서는 맞지 않다고 가정하고, 항상 함수를 거쳐 값을 네트워크 바이트 순서로 바꾸면 됩니다. 필요할 때 함수가 마법처럼 변환해 줄 것이고, 이러면 엔디언이 다른 기계에서도 코드가 이식 가능합니다.

좋습니다. 변환할 수 있는 숫자는 두 종류입니다: short(2바이트)와 long(4바이트). 이 함수들은 unsigned 변형에도 적용됩니다. 예를 들어 short를 호스트 바이트 순서에서 네트워크 바이트 순서로 바꾸고 싶다면, "host"의 h, "to", "network"의 n, "short"의 s를 붙여 htons()가 됩니다(읽을 때는 "Host to Network Short").

거의 너무 쉽죠...

정말 멍청한 조합만 아니라면 n, h, s, l을 온갖 방식으로 조합할 수 있습니다. 예를 들어 stolh()("Short to Long Host") 같은 함수는 없습니다. 적어도 여기서는요. 하지만 다음 함수들은 있습니다:

함수 설명
htons() 호스트에서 네트워크 short로
htonl() 호스트에서 네트워크 long으로
ntohs() 네트워크에서 호스트 short로
ntohl() 네트워크에서 호스트 long으로

기본적으로 숫자가 네트워크로 나가기 전에는 네트워크 바이트 순서로 바꾸고, 네트워크에서 들어올 때는 호스트 바이트 순서로 바꾸면 됩니다.

소켓 API에는 표준 64비트 변환 함수가 없지만, 다른 선택지는 htons() 참고 페이지에서 이야기합니다. 부동소수점을 다루고 싶다면 훨씬 아래에 있는 직렬화 절을 보세요.

별다른 언급이 없다면 이 문서의 숫자는 호스트 바이트 순서라고 가정하세요.

3.3 구조체

드디어 여기까지 왔습니다. 이제 프로그래밍 이야기를 해 봅시다. 이 절에서는 소켓 인터페이스에서 사용하는 여러 데이터 형을 살펴볼 텐데, 그중 일부는 이해하기가 꽤 까다롭습니다.

먼저 쉬운 것부터 보죠. 소켓 디스크립터입니다. 소켓 디스크립터의 타입은 다음과 같습니다:

int

그냥 평범한 int입니다.

이제부터 조금 이상해지니, 차분히 읽어 주세요.

나의 첫 번째 구조체(TM), struct addrinfo입니다. 이 구조체는 비교적 최근에 등장했으며, 이후에 사용할 소켓 주소 구조체를 준비하는 데 쓰입니다. 호스트 이름 조회와 서비스 이름 조회에도 사용됩니다. 실제 사용 예를 보면 더 와닿겠지만, 지금은 연결을 만들 때 가장 먼저 호출하는 것 중 하나라고만 알아두면 됩니다.

struct addrinfo {
    int              ai_flags;     // AI_PASSIVE, AI_CANONNAME, etc.
    int              ai_family;    // AF_INET, AF_INET6, AF_UNSPEC
    int              ai_socktype;  // SOCK_STREAM, SOCK_DGRAM
    int              ai_protocol;  // use 0 for "any"
    size_t           ai_addrlen;   // size of ai_addr in bytes
    struct sockaddr *ai_addr;      // struct sockaddr_in or _in6
    char            *ai_canonname; // full canonical hostname

    struct addrinfo *ai_next;      // linked list, next node
};

이 구조체에 값을 조금 채운 다음 getaddrinfo()를 호출하면, 필요한 정보가 모두 채워진 새로운 연결 리스트의 포인터를 돌려줍니다.

ai_family 필드에서 IPv4나 IPv6를 강제할 수도 있고, AF_UNSPEC으로 두어 아무거나 쓰게 할 수도 있습니다. 이런 점이 좋은 이유는 코드가 IP 버전에 종속되지 않게 해 주기 때문입니다.

이것이 연결 리스트라는 점에 주목하세요. ai_next는 다음 원소를 가리키며, 선택할 수 있는 결과가 여러 개일 수 있습니다. 저는 동작하는 첫 번째 결과를 쓰겠지만, 여러분의 요구 사항은 다를 수도 있겠죠. 제가 세상 모든 걸 아는 건 아니니까요!

struct addrinfo의 ai_addr 필드는 struct sockaddr에 대한 포인터입니다. 이제부터 IP 주소 구조체 내부의 디테일한 내용으로 들어가게 됩니다.

보통은 이런 구조체에 직접 값을 쓸 필요가 없습니다. 대개는 getaddrinfo()가 struct addrinfo를 채워 주기만 하면 충분하죠. 다만 값을 꺼내려면 이 struct들 안을 들여다봐야 하므로, 여기서 소개해 두는 것입니다.

(또한 struct addrinfo가 생기기 전에 작성된 코드에서는 이걸 전부 손으로 채워 넣었습니다. 그래서 예전 IPv4 코드에는 여전히 그런 방식이 많이 보입니다. 이 가이드의 옛 버전도 그랬고요.)

어떤 struct는 IPv4 전용이고, 어떤 것은 IPv6 전용이며, 어떤 것은 둘 다에 쓰입니다. 각각 무엇에 해당하는지 메모를 붙이겠습니다.

어쨌든 struct sockaddr는 여러 종류의 소켓에 대한 소켓 주소 정보를 담습니다.

struct sockaddr {
    unsigned short    sa_family;    // address family, AF_xxx
    char              sa_data[14];  // 14 bytes of protocol address
};

sa_family는 여러 값을 가질 수 있지만, 이 문서에서 다루는 경우에는 AF_INET(IPv4) 또는 AF_INET6(IPv6)입니다. sa_data에는 소켓의 목적지 주소와 포트 번호가 들어 있습니다. 하지만 주소를 sa_data에 손으로 하나하나 채워 넣고 싶진 않으니 꽤 다루기 불편합니다.

struct sockaddr를 다루기 위해 프로그래머들은 병행 구조체인 struct sockaddr_in을 만들었습니다. (in은 Internet를 뜻합니다.) 이것은 IPv4에 사용됩니다.

그리고 여기가 핵심입니다. struct sockaddr_in 포인터는 struct sockaddr 포인터로 캐스팅할 수 있고, 그 반대도 됩니다. 그래서 connect()가 struct sockaddr*를 원하더라도 struct sockaddr_in을 쓰다가 마지막 순간에 캐스팅하면 됩니다!

// (IPv4 only--see struct sockaddr_in6 for IPv6)

struct sockaddr_in {
    short int          sin_family;  // Address family, AF_INET
    unsigned short int sin_port;    // Port number
    struct in_addr     sin_addr;    // Internet address
    unsigned char      sin_zero[8]; // Same size as struct sockaddr
};

이 구조체는 소켓 주소의 각 원소를 참조하기 쉽게 해 줍니다. sin_zero는 구조체 길이를 struct sockaddr와 맞추기 위해 넣은 패딩이므로 memset()으로 전부 0으로 채워야 합니다. 또 sin_family는 struct sockaddr의 sa_family에 해당하므로 "AF_INET"으로 설정해야 합니다. 마지막으로 sin_port는 네트워크 바이트 순서여야 합니다(htons()를 사용하세요!).

좀 더 깊이 들어가 봅시다. sin_addr 필드는 struct in_addr입니다. 이건 대체 뭘까요? 너무 과장하고 싶진 않지만, 역대 가장 무서운 union 중 하나입니다:

// (IPv4 only--see struct in6_addr for IPv6)

// Internet address (a structure for historical reasons)
struct in_addr {
    uint32_t s_addr; // that's a 32-bit int (4 bytes)
};

와우. 예전에는 정말 union이었지만, 이제는 그런 시대가 지나간 듯합니다. 참 다행이죠. 따라서 ina를 struct sockaddr_in 타입으로 선언했다면, ina.sin_addr.s_addr는 4바이트 IP 주소(네트워크 바이트 순서)를 가리킵니다. 설령 여러분 시스템이 아직도 끔찍한 union 형태의 struct in_addr를 쓰더라도, 위에서 본 것과 똑같은 방식으로 4바이트 IP 주소를 참조할 수 있습니다(#define 덕분입니다).

IPv6는 어떨까요? 비슷한 struct가 여기에 있습니다:

// (IPv6 only--see struct sockaddr_in and struct in_addr for IPv4)

struct sockaddr_in6 {
    u_int16_t       sin6_family;   // address family, AF_INET6
    u_int16_t       sin6_port;     // port, Network Byte Order
    u_int32_t       sin6_flowinfo; // IPv6 flow information
    struct in6_addr sin6_addr;     // IPv6 address
    u_int32_t       sin6_scope_id; // Scope ID
};

struct in6_addr {
    unsigned char   s6_addr[16];   // IPv6 address
};

IPv6도 IPv4처럼 IPv6 주소와 포트 번호를 가집니다.

또 지금은 IPv6 플로우 정보와 Scope ID 필드는 설명하지 않겠습니다. 이건 어디까지나 입문 가이드니까요. :-)

마지막으로, IPv4와 IPv6 구조체를 모두 담을 만큼 충분히 크게 설계된 또 하나의 단순한 구조체 struct sockaddr_storage가 있습니다. 어떤 호출에서는 struct sockaddr에 IPv4 주소가 채워질지 IPv6 주소가 채워질지 미리 알 수 없을 때가 있습니다. 이럴 때는 struct sockaddr와 아주 비슷하지만 더 큰 이 병행 구조체를 넘기고, 나중에 필요한 타입으로 캐스팅하면 됩니다:

struct sockaddr_storage {
    sa_family_t  ss_family;     // address family

    // all this is padding, implementation specific, ignore it:
    char      __ss_pad1[_SS_PAD1SIZE];
    int64_t   __ss_align;
    char      __ss_pad2[_SS_PAD2SIZE];
};

중요한 점은 ss_family 필드로 주소 패밀리를 볼 수 있다는 것입니다. 여기서 AF_INET인지 AF_INET6인지(IPv4인지 IPv6인지) 확인한 다음, 필요하면 struct sockaddr_in 또는 struct sockaddr_in6로 캐스팅하면 됩니다.

3.4 IP 주소, 두 번째 이야기

다행히 IP 주소를 다루게 해 주는 함수가 많이 있습니다. 일일이 손으로 계산해서 << 연산자로 long 안에 집어넣을 필요가 없습니다.

먼저 struct sockaddr_in ina가 있고, 여기에 10.12.110.57 또는 2001:db8:63b3:1::3490 같은 IP 주소를 저장하고 싶다고 해 봅시다. 이때 사용하는 함수가 inet_pton()입니다. 숫자와 점 표기법의 IP 주소를, AF_INET를 주면 struct in_addr로, AF_INET6를 주면 struct in6_addr로 변환합니다. (pton은 "presentation to network"의 약자이며, 더 쉽게 기억하려면 "printable to network"라고 생각해도 됩니다.) IPv4와 IPv6 변환은 다음과 같습니다:

struct sockaddr_in sa;   // IPv4
struct sockaddr_in6 sa6; // IPv6

inet_pton(AF_INET, "10.12.110.57", &(sa.sin_addr));
inet_pton(AF_INET6, "2001:db8:63b3:1::3490", &(sa6.sin6_addr));

(짧은 메모: 예전에는 inet_addr() 또는 inet_aton()이라는 함수를 사용했습니다. 지금은 둘 다 폐기되었고 IPv6에서는 동작하지 않습니다.)

위 코드 조각은 에러 검사가 없어서 그다지 견고하지 않습니다. inet_pton()은 오류 시 -1을 반환하고, 주소 형식이 잘못됐을 때는 0을 반환합니다. 그러니 사용하기 전에 결과가 0보다 큰지 꼭 확인하세요!

이제 문자열 형태의 IP 주소를 이진 표현으로 바꿀 수 있습니다. 그 반대는 어떨까요? struct in_addr를 점-숫자 표기법으로 출력하고 싶다면요? (또는 struct in6_addr를, 뭐랄까, "16진수와 콜론 표기법"으로 출력하고 싶다면요.) 이 경우에는 inet_ntop() 함수를 사용합니다. (ntop은 "network to presentation"의 약자이며, 더 쉽게 기억하려면 "network to printable"라고 생각해도 됩니다.) 예시는 다음과 같습니다:

// IPv4:

char ip4[INET_ADDRSTRLEN];  // space to hold the IPv4 string
struct sockaddr_in sa;      // pretend this is loaded with something

inet_ntop(AF_INET, &(sa.sin_addr), ip4, INET_ADDRSTRLEN);

printf("The IPv4 address is: %s\n", ip4);


// IPv6:

char ip6[INET6_ADDRSTRLEN]; // space to hold the IPv6 string
struct sockaddr_in6 sa6;    // pretend this is loaded with something

inet_ntop(AF_INET6, &(sa6.sin6_addr), ip6, INET6_ADDRSTRLEN);

printf("The address is: %s\n", ip6);

함수를 호출할 때는 주소 타입(IPv4 또는 IPv6), 주소 자체, 결과를 담을 문자열 포인터, 그리고 그 문자열의 최대 길이를 넘깁니다. (가장 긴 IPv4 또는 IPv6 주소를 담는 데 필요한 문자열 크기는 각각 INET_ADDRSTRLEN과 INET6_ADDRSTRLEN 매크로에 들어 있습니다.)

(다시 한번 예전 방식을 말하자면, 역사적으로는 이 변환에 inet_ntoa() 함수를 사용했습니다. 이 역시 폐기되었고 IPv6에서는 동작하지 않습니다.)

마지막으로, 이 함수들은 숫자 형태의 IP 주소에만 동작합니다. www.example.com 같은 호스트 이름에 대해 DNS 네임서버 조회를 하지는 않습니다. 그런 경우에는 나중에 보게 될 getaddrinfo()를 사용합니다.

3.4.1 사설(또는 분리된) 네트워크

많은 곳에서는 외부로부터 네트워크를 보호하기 위해 방화벽으로 네트워크를 숨깁니다. 그리고 종종 방화벽은 네트워크 주소 변환(NAT)이라는 과정을 통해 "내부" IP 주소를 "외부" IP 주소(세상 사람들이 아는 주소)로 바꿉니다.

벌써 긴장되나요? "이상한 얘기만 계속하는데 대체 어디로 가는 거지?"

하지만 초보자라면 안심하고 음료 한 잔 하세요. NAT는 투명하게 처리되므로 굳이 신경 쓸 필요가 없습니다. 다만 여러분이 보는 네트워크 번호 때문에 혼란스러워질까 봐 방화벽 뒤의 네트워크 이야기를 꺼낸 겁니다.

예를 들어 저는 집에 방화벽이 있습니다. DSL 회사에서 정적 IPv4 주소 두 개를 받았지만, 네트워크에는 컴퓨터가 일곱 대 있습니다. 이게 어떻게 가능할까요? 한 IP 주소를 두 컴퓨터가 공유하면 데이터가 어디로 가야 할지 모를 텐데요!

답은 이렇습니다. 같은 IP 주소를 쓰는 게 아닙니다. 이 컴퓨터들은 2400만 개의 IP 주소가 할당된 사설 네트워크에 있습니다. 전부 제 것인 셈이죠. 적어도 다른 사람들 입장에서는 그렇습니다. 실제로는 이런 일이 벌어집니다:

제가 원격 컴퓨터에 로그인하면, 그 컴퓨터는 제가 192.0.2.33에서 로그인했다고 말합니다. 이건 ISP가 제게 제공한 공인 IP 주소입니다. 하지만 제 로컬 컴퓨터에 자기 IP가 뭐냐고 물으면 10.0.0.5라고 합니다. 한 주소를 다른 주소로 바꾸는 건 누구일까요? 맞습니다, 방화벽입니다. NAT를 하는 거죠!

10.x.x.x는 완전히 분리된 네트워크나 방화벽 뒤 네트워크에서만 사용하도록 예약된 몇 안 되는 네트워크 중 하나입니다. 사용 가능한 사설 네트워크 번호에 대한 자세한 내용은 RFC 191819에 나와 있지만, 흔히 보는 것은 10.x.x.x와 192.168.x.x입니다. 여기서 x는 보통 0에서 255 사이이고, 조금 덜 흔한 것으로 172.y.x.x가 있는데 y는 16에서 31 사이입니다.

NAT를 수행하는 방화벽 뒤의 네트워크가 반드시 이런 예약 네트워크를 써야 하는 것은 아니지만, 보통은 그렇습니다.

(재밌는 사실! 제 외부 IP 주소는 실제로 192.0.2.33이 아닙니다. 192.0.2.x 네트워크는 이 가이드처럼 문서에서 사용할 가짜 "실제" IP 주소용으로 예약돼 있습니다. 놀랍죠!)

어떤 의미에서는 IPv6에도 사설 네트워크가 있습니다. RFC 419320에 따르면 주소는 fdXX:(혹은 미래에는 fcXX:)로 시작합니다. 다만 NAT와 IPv6는 일반적으로 잘 섞이지 않습니다(이 문서 범위를 벗어나는 IPv6-to-IPv4 게이트웨이 같은 경우가 아니라면요). 이론상 여러분은 NAT가 더 이상 필요 없을 만큼 많은 주소를 갖게 되기 때문입니다. 그래도 외부로 라우팅되지 않는 네트워크에서 스스로 주소를 할당하고 싶다면 이 방법을 쓰면 됩니다.
